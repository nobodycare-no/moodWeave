# MoodWeave - AI Agent 多智能体协作协议

> 版本：v2.1
> 本文件是 AI 与 AI 之间协作的核心规范。任何 AI 开始工作前必须先读本文件。

---

## 1. 身份识别

每个 AI 通过 Git 身份标识自己，commit 历史就是身份名片。

| 身份 | Git Author | 场景 |
|------|-----------|------|
| **SOLO**（家用） | `SOLO <solo@moodweave.dev>` | Home 电脑 |
| **Agent-C** | `Agent-C <agent-c@moodweave.dev>` | 公司电脑 |

---

## 2. 核心工作流

```text
用户说“继续 MoodWeave”
        ↓
1. git pull
2. 身份识别：git log --oneline -3
3. 交接判断：读 status.md + HANDOFF.md + REVIEW.md
4. 按需审查最近一次交付
5. 开始开发
```

```text
用户说“今天结束了”
        ↓
1. 更新 status.md
2. 必要时更新 HANDOFF.md
3. git add . && git commit
4. git push
```

## 3. 开工协议

用户说“继续 MoodWeave”时，必须执行以下步骤：

### Step 1: git pull

```bash
git pull
```

- 先拉取远端最新提交。
- 若出现冲突，进入第 7 节处理流程。
- 若网络失败，重试一次；仍失败则明确告知当前无法同步远端。

### Step 2: 身份识别

```bash
git log --oneline -3
```

识别最近提交里的 `[SOLO]` 或 `[Agent-C]` 标记，判断：

- 最近一次交付是谁完成的？
- 最近一次交付是否由同一身份连续完成？

### Step 3: 交接判断

读取以下文件：

- `docs/superpowers/status.md`
- `docs/superpowers/HANDOFF.md`
- `docs/superpowers/REVIEW.md`

### Step 4: 交付范围判定

> 审查不是按“最后一次改了什么文件”判断，而是按“上一次交付的完整提交范围”判断。

若最近一次提交包含以下任意内容，就视为一次完整交付，接手方必须审查：

- 代码文件
- 文档文件
- 配置文件
- 状态文件
- 交接文件

只有当最近一次提交完全是同一身份的纯文档收尾，且未引入任何代码、配置或行为变化时，接手方才可以不做交叉审查。

### Step 5: 交叉审查规则

| 情况 | 行为 |
|------|------|
| 上次提交是**对方**完成的 | 审查对方最近一次完整交付，并写入 `REVIEW.md` |
| 上次提交是**自己**完成的，且是连续同身份开发 | 可跳过审查 |
| 上次提交是**自己**完成的，但本次接手点之前存在对方交付 | 仍需审查对方那次交付 |
| 上次提交包含代码 + 文档更新 | 一律视为代码交付，不可因为“最后只改了文档”而跳过审查 |

### Step 6: 开发

进入编码阶段。

## 4. 收尾协议

用户说“今天结束了”时，必须执行：

### Step 1: 更新 `status.md`

写清楚本次收尾的**完整交付范围**，包括：

- 主要完成了什么功能
- 代码改动涉及哪些模块
- 是否包含文档、配置、流程文件

建议格式：

```markdown
### 最新更新
- [日期] [身份] 完成 XX 功能，涉及 A/B/C 模块；若同步更新了 status、REVIEW、HANDOFF，也一并写明。
```

### Step 2: 更新 `HANDOFF.md`

以下情况必须写入 `HANDOFF.md`：

- 当前步骤只完成了一半
- 需要对方继续接着做
- 有需要明确提醒对方的风险、待办或环境问题
- 本次交付包含代码与文档混合改动，且存在需要特别说明的部分

如果本次步骤完全完成且没有额外交接事项，`HANDOFF.md` 保持空文件即可。

### Step 3: git commit

```bash
git add .
git commit -m "[身份] 类型: 描述"
```

推荐格式：

```text
[SOLO] feat: 完成画布缩放和平移
[Agent-C] fix: 修复图片上传后位置偏移
[SOLO] docs: 补充交接说明
[Agent-C] merge: 解决 pull 冲突
```

若没有任何文件改动，则跳过 commit，并告诉用户今天没有代码变更。

### Step 4: git push

```bash
git push
```

push 成功后，必须明确告诉用户已同步到远端。

---

## 5. 冲突处理

`git pull` 冲突时：

1. 不要慌。
2. 优先保留双方的有效信息。
3. `status.md`、`HANDOFF.md`、`REVIEW.md` 这类文件里，只要不是重复或明显废弃的内容，默认都应保留。
4. 解决后：

```bash
git add .
git commit -m "[身份] merge: 解决冲突"
git push
```

---

## 6. 审查原则

### 6.1 审查对象

审查的对象是“最近一次完整交付”，不是“最近一次只改了什么文件”。

### 6.2 审查触发条件

接手方必须审查对方交付，除非同时满足以下条件：

- 最近一次提交是自己完成的
- 且自己之后没有出现对方的新交付
- 且本次接手点与上次自己交付属于连续同身份开发

### 6.3 审查范围

审查时至少查看：

```bash
git show <commit-hash> --stat
git show <commit-hash>
```

并核对：

- 功能是否符合 `status.md` 里的规划
- 代码风格是否合理
- 是否存在明显 bug、回归或可优化点
- 文档改动是否与代码改动一致

### 6.4 记录方式

审查结果必须写入 `docs/superpowers/REVIEW.md`，格式要包含：

- 日期
- 审查者身份
- 被审查 commit
- 审查结论
- 必要时的修改意见

## 7. 用户完全不操心的自动化

| 用户行为 | 你自动做的事 |
|---------|------------|
| “继续 MoodWeave” | `git pull` → 身份识别 → 交付判定 → 审查（如需） → 读进度 → 开发 |
| “今天结束了” | 更新 `status.md` → 必要时更新 `HANDOFF.md` → commit → push |
| 用户什么都没说，但你正在开发中 | 只做开发，不主动 Git 操作 |
| `git pull` 发生冲突 | 自动解决 → commit merge → push |

## 9. 关键修正规则

以下规则高于所有经验性判断：

1. **文档收尾不等于没有代码交付。** 如果一个提交里既有代码又有文档，接手方不能因为最后看的文件像是文档就跳过审查。
2. **审查按“交付”判断，不按“文件类型”判断。**
3. **交接记录必须写清本次交付范围。** 否则下一位 agent 很容易误判。
4. **同身份连续开发可以少审查，但不能吞掉对方的最近交付。**

## 10. 已知流程风险与修复

### 风险 1：最后一次修改是文档，导致漏审代码

错误做法：看到最近一次提交或最近一次文件修改是 `status.md`，就判断“只有文档变化，无需审查”。

正确做法：查看完整 commit diff。只要同一次提交里有代码、配置或运行行为变化，就必须按代码交付审查。

### 风险 2：多个小提交组成一次交付，但只审最后一个提交

错误做法：只审 `git log -1`。

正确做法：如果 `status.md` 或 `HANDOFF.md` 表示上次完成的是一个步骤，应审查该步骤相关的完整提交范围。至少查看最近一次对方提交；必要时扩展到 `git log --oneline --author=<对方> -5`。

### 风险 3：状态文件被当作真实事实，而不校验代码

错误做法：`status.md` 写了“已完成”，就直接进入下一步。

正确做法：状态文件只是索引。开始下一步前，应快速核对对应文件是否存在、构建是否能跑、上一步是否真的落地。

### 风险 4：网络失败时继续开发但没说明

错误做法：`git pull` 失败后直接开发，最后假设远端没变化。

正确做法：重试一次；若仍失败，明确记录“未能同步远端”，并在最终回复里提醒用户。

### 风险 5：HANDOFF 长期为空，下一位 agent 只能猜上下文

错误做法：只更新 `status.md`，不说明本次交付范围。

正确做法：步骤完整完成且无风险时可以保持 `HANDOFF.md` 为空；但凡有未完成、混合改动、环境问题、网络失败、测试未跑，都必须写 `HANDOFF.md`。
